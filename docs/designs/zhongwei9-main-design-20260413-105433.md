# Design: 智能文档拆分与版本化管理

## Problem Statement

当前的 Doc2Know 是一对一转换：一个 Word/PDF 文档生成一个 Markdown 文件。

用户希望实现智能拆分：一个输入文档能自动分解成多个按主题分类的输出文档。例如一份 100 页的产品手册，能拆分成：
- `安装指南.md`
- `功能教程.md`
- `故障排除.md`
- `API 参考.md`

## What Makes This Cool

**核心价值**：把"线性阅读"的长文档变成"按需查阅"的主题文档。

想象一下：你有一份 200 页的官方文档，现在想找"如何配置 SSL"的内容。原来要翻目录、找页码、在 PDF 里搜索。有了这个功能，直接打开 `安全配置.md` 就能找到。

最酷的是：每次生成的分类都是"全新的"——LLM 会根据对文档的理解，动态决定最佳分类方式。这不是模板替换，是真正的智能分析。

## Constraints

1. **开源项目** —— 代码要清晰、可维护，方便社区贡献
2. **开发者自用** —— 面向技术人员整理文档，不需要太花哨的交互
3. **大类优先** —— 初期分类数量要控制（3-5 个大类），不要拆太碎
4. **版本隔离** —— 每次生成结果要放在独立的版本目录中，不覆盖历史

## Premises

1. **LLM 能准确理解文档主题结构** —— 基于当前 LLM 能力，可以合理假设它能提取主要内容主题
2. **用户能接受不确定性** —— 每次生成的分类名称和数量可能不同，这是特性不是 bug
3. **两阶段分析比单阶段更可控** —— 先提取所有细粒度主题，再合并成大分类，比直接让 LLM 生成"3-5 个分类"更稳定
4. **版本目录管理是简单有效的隔离方案** —— `output/2025-01-15-v1/` 比数据库或 git 更轻量
5. **中文文件名现代系统都支持** —— 不需要拼音转换，直接用中文作为文件名显示名

## Approaches Considered

### Approach A: 最小可行版本
**Summary**: 修改现有 analyzer.py，让 LLM 直接输出多个文档的 JSON 结构。
**Effort**: S
**Risk**: Low
**Pros**:
- 最快出结果，改动最小
- 能快速验证想法
**Cons**:
- 提示词工程较简单
- 分类质量依赖单次 LLM 调用
- 无法控制分类粒度
**Reuses**: 复用现有配置系统和文件生成逻辑

### Approach B: 完整重构版本（推荐）
**Summary**:
- 全新设计 `DocumentSplitter` 类，专注于智能拆分
- 两阶段分析：先提取所有主题 → 再合并成大分类
- 版本管理：自动生成 `output/2025-01-15-v1/` 目录结构
- 配置化：支持 `split_strategy: conservative|aggressive` 控制粒度
**Effort**: M
**Risk**: Med
**Pros**:
- 架构清晰，可扩展性强
- 能精细控制分类粒度（conservative=少分类，aggressive=多分类）
- 版本隔离满足"每次全新生成"的需求
**Cons**:
- 需要更多开发和测试时间
- 两阶段分析会增加 LLM 调用次数（3 次调用：提取主题、合并分类、提取内容）
- Token 消耗约为单阶段的 2-3 倍
**Reuses**: 复用配置加载、LLM 客户端封装

### Approach C: 交互式预览版本
**Summary**:
- 先生成 `split_plan.json` 展示 LLM 建议的分类结构
- 用户确认或调整后再执行实际拆分
**Effort**: L
**Risk**: High
**Pros**:
- 用户可控性最强
- 避免"黑盒"感
**Cons**:
- 开发复杂，交互设计需要反复打磨
- 与"开发者自用"的目标不太匹配
**Reuses**: 复用现有 CLI 框架

## Recommended Approach

选择 **方案 B：完整重构版本**

**理由**：
1. 用户明确要求"完全重写分析逻辑"，方案 B 的架构最符合这个要求
2. 两阶段分析（提取主题 → 合并分类）正好能解决"大类优先"的需求
3. 版本目录管理符合"每次全新生成、历史隔离"的需求
4. 配置化的 `split_strategy` 给未来留下了灵活空间

## 架构设计详解

### 三阶段分析流程

实际实现是**三阶段**而非两阶段：

```
文档文本 → [阶段1: 主题提取] → 细粒度主题列表
                        ↓
细粒度主题 → [阶段2: 分类合并] → 大分类定义（3-5个）
                        ↓
分类定义+原文 → [阶段3: 内容提取] → 多个Markdown文件
```

**阶段 1：细粒度主题提取**
- **输入**：完整文档文本（或分块后的文本）
- **LLM 任务**：识别文档中所有独立的内容主题，不合并、不归类
- **输出格式**：JSON 数组，每个主题包含 `topic_id`, `topic_name`, `related_sections`（原文位置引用）
- **示例输出**：`[{"topic_id": "t1", "topic_name": "安装步骤", "sections": ["1.1", "3.2"]}, ...]`

**阶段 2：主题合并与分类**
- **输入**：阶段 1 的主题列表
- **LLM 任务**：将细粒度主题合并成 3-5 个大分类（conservative 模式）
- **合并策略**：基于主题名称的语义相似度，由 LLM 判断哪些主题属于同一类别
- **输出格式**：JSON 对象，`categories` 数组，每个分类包含 `category_name` 和 `topic_ids` 列表
- **示例输出**：`{"categories": [{"name": "安装指南", "topics": ["t1", "t5"]}, ...]}`

**阶段 3：分类内容提取**
- **输入**：原始文档 + 阶段 2 的分类定义
- **LLM 任务**：为每个分类提取相关内容，生成完整的 Markdown 结构
- **输出**：每个分类一个 JSON，包含 `title`, `content`, `source_refs`
- **处理方式**：对每个分类独立调用 LLM，可并行执行

### split_strategy 定义

| 策略 | 分类数量 | 适用场景 | 实现方式 |
|------|---------|---------|---------|
| `conservative` | 3-5 个 | 大类优先，适合入门文档 | 阶段 2 提示词明确约束 "合并为 3-5 个主要类别" |
| `aggressive` | 6-10 个 | 详细拆分，适合参考手册 | 阶段 2 提示词明确约束 "拆分为 6-10 个专题" |

**实现细节**：通过提示词工程控制，而非后期过滤。阶段 2 的 prompt 根据策略不同而变化。

## Open Questions（已解答）

**Q1: 分类合并策略**
**A**: 由 LLM 在阶段 2 进行语义判断。提示词设计为："分析以下主题列表，将语义相关的主题合并为 3-5 个主要类别"。LLM 的语义理解能力足以判断"安装步骤"和"安装注意事项"属于同一类别。不引入额外的相似度算法，保持架构简单。

**Q2: 内容归属冲突**
**A**: **物理合并**策略。同一主题在文档中多处出现时，阶段 3 的内容提取会汇总所有相关内容到同一个输出文件。不保留原文档的线性结构，完全按主题重新组织。如果某段内容涉及多个主题，LLM 判断其主要归属，必要时允许内容重复出现在多个分类中。

**Q3: 文件命名**
**A**: **直接用中文文件名**。现代文件系统（macOS、Linux、Windows 10+）都支持中文文件名。slug 转换只需做简单的特殊字符替换（如 `:` `*` `?` 等非法字符替换为 `-`）。不引入拼音库，避免额外依赖和可读性损失。

## 错误处理与降级策略

### 阶段 1 失败（主题提取）
- **原因**：LLM 返回格式错误、文档过大导致 chunk 处理失败
- **降级**：回退到原始的一对一转换模式，生成单个 Markdown 文件
- **日志**：记录错误详情，提示用户"智能拆分失败，已生成单文件版本"

### 阶段 2 失败（分类合并）
- **原因**：LLM 无法确定合理的分类、返回的分类数量超出约束
- **降级**：使用阶段 1 的细粒度主题直接作为分类（可能生成 10+ 个小文件）
- **日志**：警告用户"分类合并失败，使用原始主题作为分类"

### 阶段 3 失败（内容提取）
- **原因**：单个分类内容提取失败（如 token 超限）
- **降级**：跳过该分类，继续处理其他分类
- **日志**：记录失败的分类名称，提示用户"部分分类生成失败"

### JSON 解析失败
- **策略**：使用现有的 `_parse_json_response` 方法（正则提取、清理重试）
- **兜底**：如果所有解析尝试失败，将该阶段结果标记为失败，触发相应降级策略

## 性能与成本考量

### Token 消耗估算

以 100 页（约 50,000 字）的技术文档为例：

| 阶段 | 输入 Token | 输出 Token | 预估成本 (GPT-4o) |
|------|-----------|-----------|------------------|
| 阶段 1 | ~15,000 | ~2,000 | ~$0.06 |
| 阶段 2 | ~3,000 | ~1,000 | ~$0.015 |
| 阶段 3 (每个分类) | ~5,000 | ~3,000 | ~$0.05 x 4 = $0.20 |
| **总计** | - | - | **~$0.28** |

*注：阶段 3 按 4 个分类并行计算*

### 优化策略

1. **阶段 1 的 chunk 处理**：如果文档超过 `chunk_size`，每个 chunk 独立提取主题，最后合并主题列表
2. **阶段 3 并行化**：不同分类的内容提取可以并行调用 LLM，减少总耗时
3. **缓存机制（未来）**：可以缓存阶段 1 的主题提取结果，方便用户尝试不同的 split_strategy

### 预算保护

- 配置中增加 `max_total_tokens` 限制（默认 100,000）
- 超过限制时提前终止，返回已生成的部分结果

## 版本目录管理

### 命名规则

```
output/
├── 2026-04-13-1054/          # 第一次运行：日期-时间
│   ├── index.md
│   └── topics/
├── 2026-04-13-1432/          # 同一天再次运行：新时间戳
│   ├── index.md
│   └── topics/
└── latest -> 2026-04-13-1432/  # 软链接指向最新版本
```

**规则**：
- 目录名格式：`YYYY-MM-DD-HHMM`（年月日-时分）
- 自动生成，无需用户指定版本号
- 同一天多次运行会产生多个目录
- 创建 `latest` 软链接方便访问最新结果

### 清理策略

- 默认保留最近 30 天的版本
- 提供 `doc2know cleanup --days 7` 命令手动清理
- 不考虑自动删除，避免误删用户可能需要的旧版本

## Success Criteria

### 正面测试（必须全部通过）
- [ ] 能处理一份 50 页以上的测试文档
- [ ] 生成分类数量在 3-5 个之间（conservative 模式）
- [ ] 每个生成的 Markdown 文件都有合理的文件名和内容
- [ ] 版本目录正确创建，历史版本不被覆盖
- [ ] 配置文件中可以切换 conservative/aggressive 模式

### 负面测试（降级策略验证）
- [ ] 当 LLM 返回无效 JSON 时，能降级为单文件输出
- [ ] 当阶段 2 合并失败时，能使用原始主题继续生成
- [ ] 当单个分类提取失败时，不影响其他分类的生成
- [ ] 文档为空或内容过少时，给出清晰的错误提示

### 性能基准
- [ ] 100 页文档处理时间 < 2 分钟（含 3 次 LLM 调用）
- [ ] Token 消耗在预估范围内（< $0.30）
