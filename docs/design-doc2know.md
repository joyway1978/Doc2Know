# Design: Doc2Know - 文档知识库自动生成工具

Generated on 2026-04-10
Project: Doc2Know
Status: APPROVED
Mode: Builder (Open Source)

## 问题陈述

组织内部充斥着各种格式的非结构化文档（Word、PPT、Excel），这些文档存在以下问题：
- **重复冗余**：同一份信息分散在多个文档中
- **缺乏结构**：内容没有层级划分，查找困难
- **维护困难**：文档频繁过期，人工更新成本高
- **无法用于AI**：散乱的内容无法直接被AI系统（如客服机器人）使用

具体场景：一个小型客服团队（5-10人），需要1人专职维护知识库，每周都有文档过期，人工整理耗时且难以保持更新。

## 什么让这个项目很酷

1. **解决真实痛点**：不只是"整理文档"，而是把人力从重复劳动中解放出来
2. **AI驱动的智能化**：利用大模型理解文档语义，自动提取结构和关键信息
3. **开放标准输出**：生成标准的Markdown格式，可以被任何系统消费（文档站、AI客服、Wiki等）
4. **增量处理能力**：不只是单次转换，而是可持续维护的知识库工作流

## 约束条件

- 开源项目，追求简洁和可扩展性
- 支持国产大模型（通过OpenAI兼容API）
- 命令行工具形态，适合技术用户和CI/CD集成
- 初始版本只支持Word文档（.docx），后续可扩展

## 前提假设

1. **假设1**：大模型能够准确理解文档内容并提取结构化信息
2. **假设2**：用户愿意使用命令行工具，或可以将此工具集成到现有工作流
3. **假设3**：Word文档是主要的知识载体（至少在这个场景下）
4. **假设4**：Markdown格式是知识库的可接受输出格式

## 考虑的方案

### 方案A：最小可行版本（Python脚本）
**概要**：一个单文件Python脚本，读取docx，调用API，输出markdown
**工作量**：S（几天）
**风险**：低
**优点**：
- 最快验证核心想法
- 代码量最小，易于理解
- 立即能用
**缺点**：
- 扩展性差，难以添加新功能
- 没有配置管理
- 错误处理简单
**适用场景**：快速原型验证

### 方案B：模块化命令行工具（推荐）
**概要**：结构化的Python包，支持配置、日志、并发处理
**工作量**：M（1-2周）
**风险**：中
**优点**：
- 可扩展的架构，易于添加新文档类型
- 配置文件管理，适合不同场景
- 良好的错误处理和日志
- 可以发布到PyPI
**缺点**：
- 需要更多前期设计
- 代码量相对较大
**适用场景**：实用的开源工具

### 方案C：Web服务
**概要**：提供Web界面上传文档，在线查看和管理知识库
**工作量**：L（1个月+）
**风险**：高
**优点**：
- 用户体验最好，非技术用户也能用
- 可以添加协作功能
- 可以做多租户SaaS
**缺点**：
- 开发和部署复杂度高
- 需要处理安全、认证等问题
- 与"开源工具"的定位略有偏离
**适用场景**：后续可能的商业版本

## 推荐方案

**选择方案B：模块化命令行工具**

**原因**：
1. 方案A虽然快，但很快会遇到扩展瓶颈，反而浪费前期投入
2. 方案C过于复杂，偏离"先解决自己问题"的初衷
3. 方案B在工程质量和开发成本之间取得平衡，既实用又可维护
4. 命令行工具是开源社区的标准交付物，易于贡献和使用

**核心模块设计**：


## 技术栈

- **语言**：Python 3.9+
- **依赖**：
  - `python-docx`：解析Word文档（提取段落文本和样式）
  - `pandoc`：备选方案，用于保留文档层级结构
  - `openai`：LLM API调用（兼容国产模型）
  - `pyyaml`：配置管理
  - `click`：命令行接口
  - `tenacity`：重试机制
- **LLM**：支持任意OpenAI兼容API（通义千问、文心一言等）

## 关键技术决策

### 1. Word文档解析策略

**问题**：`python-docx`只能提取纯文本，无法可靠识别标题层级（Heading 1/2/3）

**解决方案**：
- **首选**：结合段落样式（paragraph.style.name）识别标题层级
- **备选**：使用 `pandoc -f docx -t markdown` 转换，再解析markdown结构
- **兜底**：让LLM从文本内容中推断结构

### 2. 超长文档分块处理

**触发条件**：文档内容超过模型上下文长度（如4000 tokens）

**分块策略**：
```
1. 按段落分割文档
2. 累积段落直到接近token限制
3. 每次调用LLM处理一个块
4. 最后汇总所有块的结构信息
```

### 3. 错误处理与重试

| 错误类型 | 处理策略 |
|---------|---------|
| API限流 | 指数退避重试（1s, 2s, 4s, 8s） |
| JSON解析失败 | 使用正则提取JSON，失败则标记需人工检查 |
| 网络超时 | 重试3次后跳过该文档，记录到error.log |
| 配置错误 | 启动时验证，给出清晰的错误提示 |

### 4. 输出文件命名规则

- 默认：使用文档标题的slug化版本（`客户常见问题` → `customer-faq.md`）
- 冲突处理：添加数字后缀（`customer-faq-1.md`）
- 自定义：支持配置模板（如 `{date}-{title}.md`）

## 核心Prompt设计

```
你是一个文档结构分析师。请将输入的文档内容分析整理成结构化格式。

要求：
1. 识别文档的主要主题和子主题
2. 提取关键信息点，去除冗余内容
3. 保持原文的准确性，不添加未提及的内容
4. 为文档生成一个简短的标题（不超过20字）
5. 生成一段摘要（不超过100字）
6. 给文档打上3-5个标签（标签由LLM根据内容自动生成）

输出必须是JSON格式：
{
  "title": "文档标题",
  "summary": "文档摘要",
  "tags": ["标签1", "标签2"],
  "sections": [
    {
      "heading": "一级标题",
      "subsections": [
        {"heading": "二级标题", "content": "内容..."}
      ]
    }
  ]
}

如果内容无法解析为结构化格式，返回：
{"error": "无法解析的原因说明", "raw_excerpt": "原文前200字"}
```

## 配置验证

启动时验证以下内容，失败给出清晰错误信息：

| 检查项 | 验证内容 | 错误提示 |
|-------|---------|---------|
| API密钥 | 格式非空且长度合理 | "API密钥未配置或格式错误，请在config.yaml中设置llm.api_key" |
| API连通性 | 发送一个test请求 | "无法连接到API，请检查base_url和api_key是否正确" |
| 模型可用性 | 验证模型名称在服务端存在 | "模型'{model}'不可用，请检查配置" |
| 目录权限 | raw_dir可读，output_dir可写 | "无法读取输入目录/无法写入输出目录" |

## API成本估算

以通义千问-turbo为例（参考价格）：

| 文档规模 | Token数 | 输入成本 | 输出成本 | 单次处理成本 |
|---------|--------|---------|---------|-------------|
| 小文档（5页） | ~3K | ¥0.003 | ¥0.006 | ~¥0.01 |
| 中文档（20页） | ~10K | ¥0.01 | ¥0.02 | ~¥0.03 |
| 大文档（100页） | ~50K | ¥0.05 | ¥0.10 | ~¥0.15 |

**100个中等规模文档的总成本约 ¥3-5**
